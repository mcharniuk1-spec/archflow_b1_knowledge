How this page works

One documented path from project context to an operator-ready package.

Start by entering the smallest useful objective and the exact evidence boundary. Knowledge Service turns that request into a local report with facts, interpretations, hypotheses, gaps, output requirements, and review questions. Agent Control can then reuse the report ID to propose roles, skills, review gates, and a file architecture for an approved operator. The initial result is always a browser-local report, never a fabricated model answer or an executed repository change.

  • Activity context is browser-local and shared with the dashboard on this device; it is not authentication or individual persistent memory.

  • The report and handoff downloads are proposals. Git, repository file creation, memory promotion, provider use, deployment, database writes, and other external writes remain separate approval gates.

Read the state labels precisely

Prepared means a browser-local report or handoff can be downloaded. Reviewed means a human accepts a bounded report outside this page. Executed is never inferred here; it needs separate runtime evidence. Written back is disabled in this public surface.

Jarvis FAQ

What do I enter? A goal, allowed evidence and exclusions, requested output, independent reviewer, and stop conditions. The dashboard manual maps each value to its versioned configuration point.

What is downloaded? A Markdown architecture report or JSON handoff that names the browser-local state and proposed file architecture. Neither download changes a repository.

What should happen next? Review the package, then use a separately approved operator workflow to make a scoped change and run its checks.
