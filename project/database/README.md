# Public catalog data model

ArchFlow currently ships a generated, public-safe JSON catalog. It is not a hosted database and it does not contain customer data, private source text, credentials, run traces, or a connection to an operator device.

The catalog exists so a person evaluating the repository can inspect the public skills, role contracts, workflow nodes, sources, and public-safe run records without having to infer the architecture from page copy.

## What is stored now

- `skill-catalog.json`: the reviewed skill packages included in this repository.
- `skill-catalog.schema.json`: the validated shape for that generated catalog.
- `role-catalog.json`: all declared public role contracts, split into packaged public skills and project method checklists.
- `role-catalog.schema.json`: the documented shape for the generated role catalog.
- `runtime-event.schema.json`: the required sanitized event shape before a future dashboard may claim live workflow state.
- `review-bundle.schema.json`: the browser-local export contract for an operator handoff.
- `project/dashboard/data.json`: the generated public console index for skills, roles, workflow nodes, sources, and run summaries.
- browser-local drafts: workflow edits and review packets saved only in the browser until the operator explicitly downloads them.

`skill-catalog.json` and `role-catalog.json` are generated by `project/scripts/generate-dashboard-data.py`. Do not edit them by hand.

## Query boundary

The dashboard Data Lab provides a read-only SQL-like teaching interface over generated JSON. It supports a small `SELECT ... FROM ... LIMIT ...` subset for these public tables:

- `skills`
- `roles`
- `workflow_nodes`
- `sources`
- `runs`

It is not a SQL server, does not execute arbitrary SQL, and cannot reach a private or production database. This deliberate limit prevents the public UI from implying a data-access capability that has not been built or secured.

## Future local database path

An installable single-user edition may add a local SQLite or DuckDB evidence ledger only after it has a schema migration path, read-only query role, query timeout, row limit, audit log, backup/restore test, and clear separation from private source corpora. A hosted multi-user database needs tenancy, authentication, RBAC, encryption, retention policy, and operational recovery proof before it can be marketed.
