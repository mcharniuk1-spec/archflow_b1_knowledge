## ICP decision

FACT:

- The owner-supplied brief prioritizes enterprise questionnaire/RFP and onboarding/hiring as the strongest immediate triggers.
- It positions key-person departure next and treats AI-agent context, repeated search, and PRD creation as weaker or less schedulable triggers.
- The narrow cohort is approximately 30-75-person product-led B2B SaaS, with Head/VP Product or CPO as the first buyer hypothesis.
- Senior PM is normally the champion; founder is a possible buyer in smaller firms; CTO is a secondary route.
- The source document has no complete embedded citation trail for its quantitative claims.

INTERPRETATION:

- “Knowledge continuity system” is a stronger category than “PRD generator.”
- The service wedge should make a real forcing moment safer and more answerable, then leave an operating system behind.
- RFP ownership must be routed through Product, Security, Sales Engineering, and RevOps rather than assuming one universal buyer.

HYPOTHESIS:

- The narrow cohort will expose higher urgency and lower procurement complexity than larger firms.
- Architecture documentation, explicit evidence boundaries, and reversible setup will reduce trust friction.
- The seven-layer tower will make the system more comprehensible than a generic feature grid.

GAP:

- No primary interview or paid-start evidence validates the ICP, buyer, channel, style, or packaging.
- The 80-100 employee transition remains unresolved.
- Willingness to grant source-system access and to pay for ongoing maintenance is unknown.

## Website implementation

### Block 1 - Seven-layer architecture tower

- Uses one exact 1024 × 1536 master image, optimized as WebP.
- Each layer is a mask over the same master, so the full tower and decomposed floors cannot drift out of registration.
- Scroll progression removes the tower from layer 7 to layer 1 and replaces the pain, outcome, and methods in the same reading position.
- Step buttons support direct navigation; reduced-motion and capture modes keep a static complete tower.
- Mobile has no horizontal overflow and maintains the explanation before the visual.

### Block 2 - Two-lane service pack

- Lane 1: Knowledge Reliability Setup - map, structure, prepare, and prove.
- Lane 2: Agent Operating System - contract, route, govern, and package.
- The previous pointer-driven tilt behavior is retained for cards and actions.
- The bridge names the reviewed knowledge spine and versioned handoff.

### Block 3 - Knowledge and agent ROI scenario

- Inputs: people, weekly context-recovery hours, blended cost, tool surfaces, centralization, governance, recoverable share, and setup scenario.
- Outputs: self-reported modeled readiness index, recovered work, modeled monthly/yearly value, and scenario payback only after a non-zero recovery assumption.
- The readiness index uses disclosed, unbenchmarked 40/40/20 heuristic weights for centralization, governance, and tool-surface pressure; it is not a measured optimization score.
- Formula and limitations are visible; values are browser-local and are not performance promises.
- The complete calculator fits in a 390 × 844 viewport without horizontal overflow.

## Dashboard and Jarvis implementation

The dashboard now uses an API-documentation information architecture:

- Overview;
- Architecture;
- Knowledge;
- Agents and Skills;
- Runs and Evidence;
- Reference;
- Workflow;
- Plan;
- a separate Jarvis page.

The architecture pages show purpose, inputs, outputs, parameters, effects, failure modes, proof, approvals, and source paths. The workflow editor keeps existing browser-local graph functions, adds compact nodes, Canvas/Stage modes, a collapsible inspector, full-screen `100dvh` operation, Escape exit, and a mobile Stage default.

Jarvis now separates four runtime states: API reachable, provider switch requested, owner authorized, and model executed. The browser can search the public OpenRouter catalog. Provider execution is deliberately blocked until all of these controls exist:

1. `MODEL_PROVIDER=openrouter`;
2. `JARVIS_ENABLE_PROVIDER_CALLS=true`;
3. server-side provider key;
4. server-side owner token;
5. explicit budget variables plus a durable projected/actual spend ledger or provider-enforced key limit;
6. a server-allowlisted model;
7. a server-issued, request/model-bound, single-use approval nonce with replay rejection.

The browser never receives the provider key. The owner token is held in memory for the tab, is not persisted, and is sent only to the same origin or HTTP loopback. Writeback remains disabled. The current acknowledgement checkbox is not represented as replay protection, and configured budget values are not represented as consumption controls.

The curated routes verified against the public catalog on 2026-07-13 are GPT-5.6 Luna, Claude Sonnet 5, Qwen3.7 Plus, and DeepSeek V4 Flash. Model IDs and pricing are temporally unstable and must be refreshed before activation: <https://openrouter.ai/api/v1/models>.

## Installable product path

E8 is a gated future milestone, not a current product claim:

- E8.0 productization decision gate;
- E8.1 GitHub-installable package and bounded source discovery;
- E8.2 least-privilege MCP server and approval-gated tools;
- E8.3 admin control plane for sources, goals, roles, permissions, evidence, approvals, drift, and receipts;
- E8.4 install, upgrade, uninstall, rollback, and operator documentation;
- E8.5 guarded pilot and benchmarked release decision.

Core user stories:

- As a product leader, I can see which source, owner, and freshness state supports an answer.
- As an operator, I can configure goals, roles, skills, budgets, retrieval, approvals, and stop rules without editing hidden prompts.
- As a reviewer, I can inspect evidence, attempts, model/usage receipts, failures, and proposed memory changes before approval.
- As an administrator, I can install, upgrade, roll back, and uninstall without leaving credentials or uncontrolled write tools.
- As an agent builder, I can generate a bounded role architecture from a parameter contract and prove it on a fixture before promotion.
