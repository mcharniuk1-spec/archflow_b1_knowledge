# Feedback And Measurement Specification

Status: report-only learning loop
Date: 2026-07-10

## Objective

Learn which public-safe explanations of the Product Discovery-to-Production PRD Pack are clear enough to justify the next planning hypothesis. This loop does not evaluate social accounts, automate distribution, or claim validated demand.

## Input Boundary

The only permissible input after owner-authorized publication is an aggregate summary supplied or approved by the owner. It must exclude names, handles, profile/account data, raw messages, screenshots, and unapproved public or private content.

## Signal Classes

| Class | Examples | Decision use |
|---|---|---|
| Attention | Aggregate impressions, saves, shares, or high-level replies | Possible clarity or format hypothesis only |
| Buyer-risk signal | A request for a diagnostic, willingness to share a redacted source packet, proposal request, or referral to a budget owner | Candidate for follow-up review; not demand proof |
| Stronger commercial evidence | Paid diagnostic interest, firm paid intent, or an approved paid start | Requires separate E6/E7 evidence and review; outside this content architecture |

## Feedback Record

Record the content-item ID, format, claim label, CTA, aggregate signal class, learning hypothesis, and next controlled experiment. Do not record account-level data. A feedback record can revise topic priority; it cannot turn a hypothesis into FACT without independently traceable evidence.

## Loop Controls

- Feedback is written only after the owner provides a safe aggregate summary.
- The content owner and AF Review decide whether a learning note changes the next calendar.
- If data quality, permission, or attribution is uncertain, record `GAP` and do not score performance.
- No automated campaign optimization, audience targeting, or external writeback is in scope.
