# ArchFlow Current State Report

Prepared: 2026-07-03
Scope: public-safe founder briefing, current architecture, knowledge-system methodology, proof evidence, and next transfer state.

## Executive Summary

ArchFlow is now organized around a single reliable operating path:

```text
approved source material
-> source boundary review
-> structured context
-> PRD and ICP output
-> task and responsibility plan
-> knowledge-base update
-> review gate
-> optional external action only after approval
```

The project has moved from a reset and documentation state into a working proof system. The current reliable surface is static/browser-local dashboard plus provider-disabled Jarvis review packets, Codex-operated execution, WikiLLM durable memory, Graphify structural reference, bounded LlamaIndex retrieval, LangGraph routing, CrewAI role structure, and public-safe Notion/Git reporting.

The system is not yet a fully autonomous production runtime. Provider-backed Jarvis generation, Railway always-on workers, dashboard-driven writeback, full vector defaulting, raw voice storage, live Nexus actions, broad external repo ingestion, and autonomous Notion/GitHub/Telegram updates remain gated.

### LlamaIndex

Role: bounded retrieval layer over approved public corpus.

Current state:
- Approved corpus: `project/`, `history/`, `skills/`, and `wiki/`.

- Safety: refuses private-source matches by default and requires source paths in output.
- Benchmark: 20-query retrieval benchmark passed with lexical recall and hybrid recall at 20 out of 20 in fallback mode.

Limit:
- The local embedding endpoint was not reachable, so vector execution fell back to lexical retrieval. Full vector defaulting remains blocked until vector availability and benchmark pass together.

## Methodology

### Evidence First

Every claim should map to one of:

- a repo file;
- a run handout;
- a WikiLLM run note;
- a workflow validation result;
- a dashboard smoke result;
- a Notion connector update;
- a sent or skipped Telegram status artifact.

### FACT / INTERPRETATION / HYPOTHESIS / GAP

The project uses this structure for meaningful decisions:

- FACT: what is proven by local state or connector output.
- INTERPRETATION: what the facts mean for architecture or operations.
- HYPOTHESIS: what may be true but still needs proof.
- GAP: what remains unproven, blocked, or approval-gated.

### Source Boundaries

Sources are routed as:

- approved public file;
- sanitized summary;
- private source for local reasoning only;
- blocked source.

The default rule is no raw private ingestion and no secrets in public files.

### Token Economy

The knowledge system avoids broad context dumps:

- read routing files first;
- use WikiLLM and run handouts before broad repo reads;
- use Graphify for structure when available;
- use LlamaIndex for approved retrieval;
- write detailed durable output into files instead of chat;
- keep chat final answers short.
