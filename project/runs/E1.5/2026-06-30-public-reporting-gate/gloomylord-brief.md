# GloomyLord Internal Reporting Brief

Date: 2026-06-30
Status: initial brief

## Role

GloomyLord is the internal visual/reporting sidecar for ArchFlow process proof. It prepares report templates, carousel structures, dashboard-friendly cards, and public-safe visual systems from approved artifacts.

## First Work Package

1. PRD for public reporting templates.
2. Research-template set for account evidence cards and process proof cards.
3. Daily and weekly monitoring templates.
4. Backlog of report ideas tied to E1-E7.
5. One-slide to six-slide carousel structure for LinkedIn-first reporting.
6. Dashboard-friendly status-card and proof-card layout rules.

## Visual Rules

- Use product-system and research-lab discipline.
- Use readable mobile typography.
- Use evidence labels, redaction bars, source-status chips, and simple process diagrams.
- Prefer artifact before/after blocks over abstract AI visuals.
- Do not use private screenshots, raw Notion screenshots, fake live dashboards, customer logos, or named target accounts.
- Do not turn internal proof into demand proof.

## Default Slide Pattern

1. Problem: messy product material or PRD handoff bottleneck.
2. Input boundary: what sources are allowed and what stays private.
3. Workflow: context digest, PRD, backlog, acceptance criteria, KB update.
4. Review gate: AF Review, source grades, owner approval.
5. Output proof: artifact paths or sanitized screenshots after approval.
6. Next question: what product teams should measure or fix next.

## Dashboard Integration

GloomyLord can design dashboard cards but cannot create live state, write actions, or publish copy. Dashboard remains read-only until the live API/worker/voice gate exists.
